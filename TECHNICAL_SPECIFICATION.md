# Техническое задание на разработку веб-интерфейса для системы автоматизации подготовки исковых заявлений

## 1. Общие сведения

### 1.1. Наименование проекта
**LegalDocInspector** - система автоматизации обработки юридических документов и генерации исковых заявлений

### 1.2. Назначение системы
Веб-интерфейс предназначен для автоматизации процесса подготовки исковых заявлений в арбитражный суд на основе загруженных документов (договоров, претензий, справок о задолженности) с использованием технологий машинного обучения и обработки естественного языка.

### 1.3. Цели разработки
- Автоматизация процесса обработки юридических документов
- Ускорение подготовки исковых заявлений
- Минимизация ошибок при заполнении документов
- Упрощение работы с множественными наборами документов

---

## 2. Технические требования

### 2.1. Технологический стек
- **Frontend Framework**: Streamlit
- **Язык программирования**: Python 3.11+
- **HTTP клиент**: requests
- **Работа с документами**: python-docx
- **Обработка данных**: pandas

### 2.2. Интеграция с бэкендом
- **Базовый URL**: `http://localhost:5001`
- **Эндпоинты**:
  - `POST /parse` - парсинг документов
  - `POST /calculate_penalty` - расчёт пени
  - `POST /create_doc` - создание иска
  - `POST /create_calculating_table` - создание расчёта к иску

### 2.3. Внешние зависимости
- Интеграция с сервисом получения данных по ИНН (`parse_info_by_inn`)
- Интеграция с калькулятором госпошлины (`calculate_state_duty`)

## 3. Функциональные требования

### 3.1. Управление состоянием приложения

#### 3.1.1. Session State
Интерфейс должен использовать Streamlit Session State для хранения:
- Данных формы (`form_data`):
  - Информация о суде (`court_info`)
  - Информация об истце (`plaintiff_info`)
  - Информация об ответчике (`defendant_info`)
  - Информация об иске (`lawsuit_info`)
  - Результаты обработки (`result`, `result2`)
  - Флаги состояния (`flag`, `flag2`, `flag3`)
  - Флаги валидации (`plaintiff_correct`, `plaintiff_uncorrect`)
  - Флаг изменений (`forms_changed`)
  - Количество наборов документов (`num_complects`)
  - Тип компании (`company_type`)
  - Дата конца просрочки (`end_date`)

- Наборов документов (`complects`):
  - Для каждого набора хранить загруженные файлы:
    - Договор (`contract_uploaded_file`)
    - Претензия (`claim_uploaded_file`)
    - Справка о задолженности (`debt_certificate_file`)

- Информации о договорах (`contracts`):
  - Для каждого договора:
    - Тип услуги (`service_type`)
    - Информация о дне начала просрочки (`overdue_date`)
    - День месяца для оплаты (`day_of_penalty`)
    - Номер пункта договора (`contract_point`)

#### 3.1.2. Обработчик изменений
Реализовать функцию `on_change_handler()`, которая:
- Устанавливает флаг `forms_changed = True` при любом изменении полей формы
- Используется как callback для всех интерактивных элементов

### 3.3. Этап 2: Отправка на обработку

#### 3.3.1. Кнопка отправки
- Текст: "Отправить на сервер"
- Отображается только при выполнении условий валидации

#### 3.3.2. Формирование запроса
При нажатии кнопки:
1. Создать словарь `files = {}` для файлов
2. Создать словарь `data = {}` для метаданных:
   - Добавить `"date"`: дата в формате `"YYYY-MM-DD"`

3. Для каждого набора документов (`complect_id`, `complect_info`):
   - Получить файлы из набора
   - Выполнить `seek(0)` для каждого файла
   - Добавить в `files`:
     - `f'complect_{complect_id}_claim_file'`: (имя файла, файл)
     - `f'complect_{complect_id}_contract_file'`: (имя файла, файл)
   - Для каждого файла справки о задолженности (поддержка множественных файлов):
     - Добавить в `files`: `f'complect_{complect_id}_certificate_file_{i}'`: (имя файла, файл)
     - Добавить в `data`: `f'{complect_id}_certificates_count'`: количество файлов

4. Добавить в `data`: `'complects_count'`: количество наборов

#### 3.3.3. Отправка запроса
- Метод: `POST`
- URL: `http://localhost:5001/parse`
- Параметры: `files=files`, `data=data`
- Отобразить спиннер с текстом: "Ваш запрос обрабатывается, пожалуйста, подождите, обработка одного набора занимает в среднем 2 минуты"

#### 3.3.4. Обработка ответа
- При `status_code == 200`:
  - Установить `flag = True`
  - Сохранить JSON ответ в `form_data['result']`
  - Сохранить `path_to_save` из ответа
- При ошибке:
  - Показать сообщение об ошибке с кодом статуса
  - Отобразить текст ответа

## 4. Требования к обработке ошибок

### 4.1. Ошибки загрузки файлов

- При ошибке получения данных по ИНН:
  - Показать сообщение об ошибке
  - Предоставить возможность повторной попытки

- Проверка минимального количества наборов (не менее 1)
- Проверка заполненности обязательных полей перед генерацией документов
