# Промт: сгенерируй демо-приложение для демонстрации архитектуры системы

Сгенерируй минимальное, но рабочее демо-приложение, которое показывает архитектуру:
- `Web UI`
- `API/BFF`
- `Message Broker` (RabbitMQ)
- `Parser Service`
- `Document Generator Service`

Допустим альтернативный клиент вместо `Web UI`:
- Python-скрипт, который отправляет HTTP-запросы в те же endpoint-ы
- и передает данные в том же формате JSON, что и веб-интерфейс.

Важно: это именно **демонстрация архитектуры**, а не полная бизнес-реализация.

---

## 1. Цель демо

Нужно продемонстрировать поток:
1) UI отправляет в API первый JSON с файлами/метаданными пакета,
2) API публикует команду парсинга в брокер,
3) `Parser Service` выполняет **имитацию обработки** (через задержки) и публикует прогресс,
4) API возвращает в UI JSON с данными, извлеченными из файлов,
5) UI отправляет обратно JSON **того же формата**, но с измененными пользователем значениями,
6) API публикует команду генерации документа в брокер,
7) `Document Generator Service` выполняет имитацию генерации и публикует прогресс/результат,
8) UI показывает прогресс в реальном времени (или polling раз в 2 секунды) и получает финальный результат.

---

## 2. Функциональные ограничения

1. Не делать реальный парсинг PDF/Excel/Docx.
2. Не делать реальную генерацию документов.
3. В сервисах вместо обработки:
   - сделать `sleep` на несколько секунд,
   - на каждом шаге отправлять сообщение о прогрессе (`10%`, `40%`, `70%`, `100%`),
   - в конце отправлять событие завершения.
4. Обработка должна быть последовательной:
   - сначала `Parser Service`,
   - затем возврат в UI JSON результата парсинга,
   - затем UI отправляет отредактированный JSON того же формата,
   - только после этого запускается `Document Generator Service`.

---

## 3. Требования к модели данных (обязательно)

Реализуй все структуры данных через `@dataclass`.

Для каждой dataclass обязательно реализовать:
- `from_json(...)`
- `to_json(...)`

Требования к сериализации:
1. Структура данных должна быть **детерминированной**.
2. Нельзя динамически добавлять новые поля из входного JSON.
3. `from_json()` должен:
   - принимать только известные поля,
   - валидировать обязательные поля,
   - отклонять неизвестные поля (ошибка/исключение).
4. `to_json()` должен возвращать JSON только с заранее определенным набором полей.

Пример минимального набора dataclass:
- `CommandMessage` (id, type, package_id, created_at, payload)
- `ProgressMessage` (id, package_id, stage, progress, status, message, timestamp)
- `ResultMessage` (id, package_id, status, artifacts, timestamp)
- `ErrorMessage` (id, package_id, code, message, timestamp)
- `UiEvent` (event_type, package_id, data, timestamp)

---

## 4. Минимальный API контракт

Сделай минимальные endpoint-ы:
- `POST /api/v1/packages` -> создать пакет, вернуть `package_id`
- `POST /api/v1/packages/{package_id}/parse` -> принять первый JSON и запустить этап парсинга
- `GET /api/v1/packages/{package_id}/parsed` -> вернуть JSON результата парсинга (для редактирования в UI)
- `POST /api/v1/packages/{package_id}/generate` -> принять отредактированный JSON того же формата и запустить генерацию
- `GET /api/v1/packages/{package_id}/events` -> получить новые события для UI

Дополнительно (опционально):
- `GET /api/v1/health`
- WebSocket endpoint для стриминга событий

Если WebSocket не реализуется, достаточно polling UI каждые 2 секунды.

---

## 5. Поведение Message Broker

Нужны минимум 2 типа сообщений:
- команды (`command.*`)
- события прогресса/результата (`event.*`)

Пример routing keys:
- `command.package.parse`
- `command.package.generate`
- `event.package.progress`
- `event.package.completed`
- `event.package.failed`

API/BFF:
- публикует отдельную команду парсинга и отдельную команду генерации,
- подписан на события от сервисов,
- складывает события в внутреннее хранилище (in-memory очередь/список) по `package_id`,
- отдает их UI через endpoint.

---

## 6. Требования к структуре проекта

Соблюдай модульную структуру, например:
- `app/ui` (или `apps/web`) — простой UI
- `app/api` — HTTP API
- `app/broker` — обертка над RabbitMQ
- `app/services/parser_service.py`
- `app/services/doc_generator_service.py`
- `app/models/messages.py` — все dataclass с `from_json()/to_json()`

---

## 7. Что должно быть в результате

1. Код каркаса, который запускается локально.
2. Два сервиса-воркера с имитацией шагов обработки через задержки.
3. Обмен через RabbitMQ.
4. Отображение прогресса в UI.
5. Как альтернатива UI: Python-скрипт-клиент, отправляющий HTTP-запросы с теми же данными и показывающий прогресс в консоли.
6. Строгие dataclass-модели с детерминированной JSON-структурой.
7. Краткий README:
   - как запустить RabbitMQ,
   - как запустить API,
   - как запустить worker-ы,
   - как открыть UI и увидеть прогресс,
   - как запустить Python-скрипт-клиент (если UI не используется).
8. bash-скрипт, запускающий всю демонстрацию целиком. Остановка демонстрации осуществляется одним нажатием ctrl+c

---

## 8. Критерии приемки

- Приложение демонстрирует архитектурный поток end-to-end.
- UI получает минимум 3 промежуточных сообщения прогресса и финальное сообщение результата.
- Все сообщения между сервисами проходят через dataclass-модели.
- `from_json()` не принимает неизвестные поля.
- `to_json()` всегда возвращает одинаковую, предсказуемую структуру.
