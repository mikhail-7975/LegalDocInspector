================================================================================
1. ПРИНЦИПЫ ИЗМЕНЕНИЙ
================================================================================

  - Разделение слоёв: API (тонкий) → Application/Services (оркестрация) → Domain → Infrastructure.
  - Единая точка конфигурации: конфиг загружается по окружению, передаётся в create_app(config=...).
  - Явные зависимости: сервисы получают зависимости через конструктор/фабрику, а не через g.*.
  - Валидация и ошибки: единые схемы запросов и централизованные обработчики исключений с JSON-ответом.
  - Абстракция хранилища: сохранение файлов и шаблонов через интерфейс (легче тесты и замена на S3 и т.п.).
  - Константы и имена: магические строки вынесены в config или constants.
  - Тестируемость: бизнес-логика не зависит от Flask; тесты разнесены по unit/integration.
  - Многопользовательность: идентификация пользователя/сессии (user_id, session_id); изоляция данных и хранилища по user_id; задания (job_id) привязаны к пользователю.
  - Долгие операции парсинга: перевод PDF в текст и извлечение данных вынесены в отдельный воркер-сервис; API принимает загрузку файлов и возвращает job_id (202), клиент опрашивает статус и забирает результат по готовности.

3. ПОТОК ЗАВИСИМОСТЕЙ И ИНИЦИАЛИЗАЦИЯ
================================================================================

  run.py:
    1. env = os.getenv("FLASK_ENV", "development")
    2. config = load_config(env)   # config/__init__.py: читает CONFIG_PATH или config/{env}.yaml
    3. app = create_app(config)
    4. app.run(host=..., port=5001, debug=config.debug_mode)

  create_app(config: AppConfig):
    1. app = Flask(__name__)
    2. app.config["APP_CONFIG"] = config
    3. storage = LocalFileStorage(config.save_data_folder)   # пути с учётом user_id при create_request_folder(user_id, job_id)
    4. queue = RqQueue(config.broker_url) (или Celery); job_store = RedisJobStore(config.job_store_url, ttl=config.job_result_ttl_seconds)
    5. table_parser = TableParser(); contract_parser = PDFContractParser(); claim_parser = PDFClaimParser(); egrul_parser = parse_egrul_certificate
    6. parse_service = ParseDocumentsService(config, storage, table_parser, contract_parser, claim_parser, egrul_parser)
    7. job_service = JobService(queue, job_store, parse_service, storage)   # постановка заданий и выдача статуса/результата
    8. penalty_service = PenaltyCalculationService(...); doc_service = DocGenerationService(...)
    9. app.config["SERVICES"] = { "parse": parse_service, "job": job_service, "penalty": penalty_service, "doc": doc_service }
    10. register_error_handlers(app); подключить middleware user_context (before_request: g.user_id из заголовка/cookie/токена)
    11. app.register_blueprint(health_bp); app.register_blueprint(parse_bp); app.register_blueprint(jobs_bp); app.register_blueprint(penalty_bp); app.register_blueprint(documents_bp)
    12. return app

  Маршрут POST /parse (многопользовательский, долгая операция в воркере):
    - Извлечь user_id из g (middleware); валидация формы и файлов.
    - Создать job_id (uuid); сохранить файлы в storage.create_request_folder(user_id, job_id), записать метаданные (form) в job_store.set_pending(job_id, user_id, meta).
    - job_service.enqueue_parse(job_id, user_id, path_to_request_folder, form_params).
    - return jsonify({ "job_id": job_id }), 202 Accepted.
  Маршрут GET /jobs/<job_id>:
    - Проверить g.user_id; job_store.get(job_id): если user_id не совпадает — 404. Вернуть { "status": "pending"|"running"|"completed"|"failed", "result": ... при completed, "error": ... при failed }.
  Маршрут GET /parse/result/<job_id> (или GET /jobs/<job_id>/result):
    - Аналогично проверка user_id; при status == completed вернуть результат (result_parser.json); иначе 202 или 404.
  Воркер (run_worker.py):
    - Загрузка config, создание queue, job_store, ParseDocumentsService (парсеры те же).
    - queue.consume(callback): при получении задания — job_store.set_running(job_id); run_parse_task(job_id) — чтение meta из job_store, вызов parse_service.parse(...) по сохранённым путям; при успехе job_store.set_completed(job_id, result), при ошибке job_store.set_failed(job_id, error).

4. КЛЮЧЕВЫЕ УЛУЧШЕНИЯ ПО МОДУЛЯМ
================================================================================

  4.1. Конфигурация (config/)
  ---------------------------
    - Отдельная папка config/ вместо configs/ (единый стиль с одной точкой входа).
    - load_config(env): выбор файла по окружению (development.yaml, production.yaml) или по CONFIG_PATH.
    - Нет жёстко прописанного пути "configs/debug_config.yaml" в коде приложения.
    - .env.example документирует переменные (CONFIG_PATH, FLASK_ENV).

  4.2. API слой (api/)
  --------------------
    - Маршруты разбиты по blueprints: health, parse, penalty, documents — проще навигация и тесты.
    - schemas/: валидация входов в одном месте; при невалидных данных — 400 с единым форматом ошибки.
    - errors.py: @app.errorhandler(Exception), @app.errorhandler(400), возврат JSON {"error": "...", "detail": "...", "code": 500}.
    - dependencies.py: единственное место, откуда маршруты получают сервисы (из app.config или g), упрощает подмену в тестах.

  4.3. Application слой (application/services/)
  --------------------------------------------
    - ParseDocumentsService: вся логика пунктов 1–11 из спецификации /parse (создание папки, цикл по комплектам, вызов парсеров, ЕГРЮЛ, сбор result_json). Маршрут только вызывает service.parse(...) и отдаёт ответ.
    - PenaltyCalculationService: цикл по parsing_results, вызов calculate_penalty и convert_data, возврат claim_data и calculator_list.
    - DocGenerationService: создание иска и расчёта по path_to_save и конфигу.
    - Плюсы: бизнес-логику можно тестировать без Flask; при смене фреймворка переносится только api/.

  4.4. Domain (domain/)
  ---------------------
    - models.py: типизированные структуры (ParseResult, DefendantInfo, ClaimData) улучшают читаемость и рефакторинг.
    - interfaces.py: IFileStorage, IParser — позволяют в тестах подменять реальные парсеры и диск на моки.

  4.5. Infrastructure (infrastructure/)
  ------------------------------------
    - storage/: LocalFileStorage сохраняет загруженные файлы и создаёт папки запроса; в тестах подменяется на InMemoryStorage или заглушку.
    - parsing/: сгруппировано по источнику (excel, pdf). Переименование exel_parser -> excel/ убирает опечатку.
    - parsing/pdf/ — единый конвейер для договора, претензии и выписки ЕГРЮЛ:
        (1) PDF → текст: реализация выбирается по типу документа (электронный PDF → Docling, скан → OCR, при необходимости — другой движок). Интерфейс IPdfToText; реализации в pdf_to_text/ (docling, ocr, ...).
        (2) Текст → данные: реализация выбирается по задаче (ключевые слова из конфига, LLM/RAG, другой способ). Интерфейс IDataExtractor; реализации в extractors/ (keyword, llm, ...).
      Конкретные парсеры (contract_parser, claim_parser, egrul_parser) собираются из комбинации pdf_to_text + extractor; можно менять способ перевода или извлечения без переписывания парсеров.
    - calculation/: penalty и adapter рядом; единый StrictFormattedMoney в shared/money.py убирает дублирование (см. DUPLICATES.md).
    - documents/: генераторы и DocxRedactor в одном месте.
    - shared/: общие утилиты (money, dates, filename_encoding) — один источник истины.

  4.6. Обратная совместимость (backend/)
  -------------------------------------
    - Пакет backend/ остаётся точкой входа для run.py (from LegalDocInspector.backend import create_app).
    - В backend/__init__.py можно оставить create_app, который внутри вызывает новую api.app.create_app(config) и регистрирует те же маршруты (через новые blueprints или старый routes.py, который вызывает сервисы).
    - Постепенная миграция: сначала ввести сервисы и вызывать их из старых routes.py; затем перенести маршруты в api/routes/ и упростить backend до реэкспорта.

  4.7. Многопользовательность и воркер-сервис (очередь заданий)
  -------------------------------------------------------------
    - Идентификация пользователя: api/middleware/user_context.py извлекает user_id (или session_id) из заголовка (X-User-Id, Authorization), cookie или JWT и записывает в g.user_id. Без авторизации можно использовать анонимный session_id (если разрешено конфигом).
    - Изоляция хранилища: LocalFileStorage.create_request_folder(user_id, job_id) создаёт каталог save_data_folder / user_id / documents_from_request_<job_id>_<timestamp>. Все операции с файлами и результатами — в рамках этого пути; доступ к результатам только при совпадении g.user_id с user_id задания.
    - Долгая операция парсинга вынесена в воркер: перевод PDF в текст (Docling/OCR) и извлечение данных занимают много времени, поэтому синхронный POST /parse не выполняет парсинг сам, а только сохраняет файлы, создаёт задание и ставит его в очередь, возвращая 202 и job_id. Отдельный процесс (run_worker.py) забирает задания из очереди (Redis/RQ или Celery), выполняет ParseDocumentsService.parse(...), сохраняет результат в job_store и обновляет статус.
    - Очередь (infrastructure/queue): интерфейс IJobQueue (enqueue(job_id, payload), consume(callback)); реализация на RQ или Celery. Payload содержит job_id, user_id, path_to_request_folder, form_params (сериализуемые), чтобы воркер мог вызвать parse без доступа к HTTP-запросу.
    - Job store (infrastructure/job_store): интерфейс IJobStore для статуса (pending → running → completed | failed) и результата (ParseResult или ошибка). Реализация на Redis с TTL или файловая (JSON по job_id). При get(job_id) возвращается user_id для проверки доступа.
    - Маршруты: POST /parse → 202 + job_id; GET /jobs/<job_id> → статус и при completed — результат (с проверкой user_id); GET /parse/result/<job_id> — то же. Клиент после 202 опрашивает GET /jobs/<job_id> до status == completed или failed.
    - Запуск: run.py — только API; run_worker.py — один или несколько воркеров (масштабирование по числу воркеров при одной очереди). Конфиг: broker_url (Redis), job_store_url, job_result_ttl_seconds.

5. ОБРАБОТКА ОШИБОК И ВАЛИДАЦИЯ
================================================================================

  - Все необработанные исключения перехватываются в errors.py: возврат JSON с полем "error" (или "message"), "detail" (traceback в debug), "code".
  - Валидация формы /parse: проверка наличия date, complects_count, egrul_certificate_file, для каждого id — complect_{id}_claim_file, complect_{id}_contract_file, {id}_certificates_count и соответствующих certificate_file_{j}. При ошибке — 400 с перечнем недостающих полей.
  - Валидация JSON /calculate_penalty: parsing_results — массив объектов с полями parsed_info, contract_number, contract_type, contract_point, day_of_penalty; company_type, end_date — обязательные. При ошибке — 400.
  - Кастомные исключения (опционально): ParseError, ValidationError, StorageError — в error handler по типу возвращать нужный код (400, 500).
  - Доступ к заданию: GET /jobs/<job_id> и GET /parse/result/<job_id> возвращают 404, если job_id не найден или user_id из g не совпадает с user_id задания (изоляция данных пользователей).

7. СВОДКА: ПОЧЕМУ ТАК ЛУЧШЕ
================================================================================

  - Чёткие слои: смена фреймворка или добавление CLI затрагивает только api/; бизнес-логика в application/ и infrastructure/ переиспользуется.
  - Тестируемость: сервисы и парсеры тестируются без поднятия Flask; в интеграционных тестах подменяются storage, queue и тяжёлые парсеры при необходимости.
  - Единая конфигурация и ошибки: меньше магических строк, предсказуемый формат ошибок для фронта.
  - Масштабируемость: добавление нового маршрута = новый blueprint + вызов сервиса; парсеры и хранилище можно менять без правок в маршрутах; воркеры можно масштабировать по числу процессов при одной очереди.
  - Один источник истины для денег, дат и кодировок имён файлов — меньше дублирования (см. DUPLICATES.md).
  - Обратная совместимость: старый run.py и импорт create_app из backend сохраняются за счёт тонкой обёртки в backend/__init__.py.
  - Многопользовательность: изоляция данных по user_id; один пользователь не видит задания и файлы другого; path_to_save и результаты привязаны к user_id и job_id.
  - Долгий парсинг не блокирует API: операция перевода PDF в текст и извлечения данных выполняется в воркере; API быстро возвращает 202 и job_id, клиент опрашивает статус и забирает результат по готовности; несколько пользователей могут одновременно отправить задания без взаимной блокировки.
