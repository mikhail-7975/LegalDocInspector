## 1.1. Назначение

### Q-1.1.1. Полный перечень типов расчётов

> **ОТВЕТ ЗАФИКСИРОВАН**

В ТЗ упоминаются «неустойка, проценты, суммы и **иные правила, задаваемые предметной областью**». Какие конкретно расчёты (помимо неустойки и процентов) должна поддерживать система на момент запуска?

**Ответ:** Система должна поддерживать **только** расчёты, реализованные в существующем калькуляторе `LegalDocInspector/legal_doc_inspector/calculator/penalty_calculator.py`. Этот калькулятор реализует **расчёт пени за просрочку коммунальных/жилищных платежей** (ст. 155 ЖК РФ) с тремя конфигурациями по типу организации:

| Тип (`company_type`) | Этапы начисления пени (ставка ЦБ = 9,5%) |
|-----------------------|------------------------------------------|
| **«УК»** (управляющая компания) | 60 дней × 1/300 → 30 дней × 1/170 → далее × 1/130 |
| **«ТСЖ»** (товарищество собственников) | 30 дней × 0% → 60 дней × 1/300 → далее × 1/130 |
| **«Прочие»** | Весь период × 1/130 |

Функциональность калькулятора включает:
- Помесячный расчёт пени по задолженности с накоплением.
- Обработка частичных погашений долга (дробление периода пени по дате оплаты).
- Обработка корректировок обязательств (годовые корректировки, доначисления).
- Определение даты начала просрочки с учётом выходных/праздничных дней (API `isdayoff.ru`).
- Поддержка «четырёхсторонних» договоров (`is_four_party`).
- Точная арифметика денежных сумм (`Decimal`, округление `ROUND_HALF_UP`).

**Архитектурное требование (Strategy / Fallback):**

Архитектура `calculation-service` должна предусматривать:

1. **Абстракцию калькулятора** — единый интерфейс (Strategy pattern), позволяющий подключать, заменять или добавлять реализации калькуляторов без изменения оркестрирующего кода.
2. **Fallback-механизм (try/except)** — если основная реализация калькулятора выбрасывает исключение, система должна попытаться выполнить расчёт альтернативной реализацией (если такая зарегистрирована) и вернуть результат с пометкой, какая реализация была использована.
3. **Выбор реализации по конфигурации договора** — возможность привязывать разные реализации калькулятора к разным типам/конфигурациям договоров (например, один калькулятор для ЖКХ-задолженности по ст. 155 ЖК, другой — для коммерческих договоров по ст. 395 ГК, если потребуется в будущем).

Схематично:
```
CalculatorRegistry
  ├─ resolve(contractType, config) → ICalculator
  │
  ICalculator (interface)
  ├─ PenaltyCalculatorV1   ← текущая реализация из penalty_calculator.py
  ├─ PenaltyCalculatorV2   ← возможная будущая реализация
  └─ ...
  
  Orchestrator:
    try:
        result = primary_calculator.calculate(data)
    except CalculationError:
        result = fallback_calculator.calculate(data)
        result.meta.usedFallback = true
```

#### Остаточные подвопросы к Q-1.1.1

- **Q-1.1.1a.** ~~Ставка 9,5% захардкожена в текущем калькуляторе. Должна ли новая система брать актуальную ключевую ставку ЦБ РФ динамически (API ЦБ / справочник) или допустимо оставить фиксированное значение с ручным обновлением?~~
  > **ОТВЕТ:** Ставка задаётся через конфигурацию (переменная окружения, конфиг-файл или аналогичный механизм). Не хардкод, не динамический запрос к API ЦБ — значение управляется при развёртывании/эксплуатации.

- **Q-1.1.1b.** ~~API `isdayoff.ru` используется для определения выходных/праздничных дней. Допустима ли внешняя зависимость на этот API в продуктивной среде, или нужен локальный справочник / кэш / fallback?~~
  > **ОТВЕТ:** Внешняя зависимость на `isdayoff.ru` допустима. Архитектурное требование: обращение к внешним ресурсам (в т.ч. `isdayoff.ru`) должно быть выделено в отдельную сущность (адаптер/сервис), доступную для подмены при отладке и покрываемую unit-тестами через мок/стаб.

### Q-1.1.2. Типы исков

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Система заточена под один конкретный тип исков (взыскание задолженности по договору) или предполагается расширение на другие категории? Если только взыскание задолженности — по каким видам договоров (поставка, аренда, подряд, оказание услуг, заём, др.)?~~

**Ответ:** Система заточена под конкретный тип исков (взыскание задолженности по коммунальным/жилищным платежам — УК, ТСЖ, Прочие). Однако в будущем допускается расширение на другие типы исков.

**Архитектурное требование:** Тип иска (включая связанные с ним калькулятор, шаблоны, правила извлечения и валидации) выделяется в **отдельный пакет** (модуль). При добавлении нового типа иска — реализуется новый пакет, без необходимости изменять ядро системы. Это согласуется с требованием Strategy/Registry из Q-1.1.1.

### Q-1.1.3. Подсудность и форматы

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Исковое заявление составляется для арбитражных судов, судов общей юрисдикции или обоих? Отличаются ли шаблоны документов в зависимости от подсудности?~~

**Ответ:** Формат заявления полностью задаётся **шаблоном Word-документа**. Заполнение шаблона выполняется существующим пакетом `LegalDocInspector/legal_doc_inspector/doc_creator/`, который содержит:

| Модуль | Назначение |
|--------|------------|
| `docx_editor.py` (`DocxRedactor`) | Низкоуровневые операции с DOCX: поиск/замена текста, клонирование параграфов и строк таблиц, слияние ячеек, управление стилями |
| `claim_generator.py` (`ClaimGenerator`) | Заполнение шаблона **искового заявления**: данные истца/ответчика, таблицы задолженности, списки договоров, претензии, правовые абзацы по типу услуги (ТЭ/ГВС/ФОТЭ) и типу компании (УК/ТСЖ/Прочие) |
| `calculation_claim_generator.py` (`CalculationClaimGenerator`) | Заполнение шаблона **расчёта к иску**: помесячная разбивка, периоды пени, формулы, итоговые таблицы |

Вопрос подсудности (арбитраж / СОЮ) определяется выбором шаблона, а не логикой кода. Если потребуется шаблон для другого суда — создаётся новый шаблон и новый пакет заполнения.

**Архитектурный принцип: 1 шаблон Word = 1 Python-пакет для заполнения.** Все генераторы работают поверх единого базового класса `DocxRedactor`. При добавлении нового шаблона реализуется новый пакет-генератор, использующий `DocxRedactor` как инфраструктуру.

### Q-1.1.4. Количество шаблонов исковых заявлений

> **ОТВЕТ ЗАФИКСИРОВАН (следует из Q-1.1.3)**

~~Сколько различных шаблонов Word (исковое заявление + расчёт к иску) предусмотрено на момент запуска? Один универсальный или несколько вариантов под разные типы договоров/подсудности?~~

**Ответ:** На момент запуска — **два шаблона** (как и в текущей реализации):

Архитектура допускает добавление новых шаблонов по принципу «1 шаблон = 1 пакет» (Q-1.1.3).

## 1.2. Границы

### Q-1.2.1. Конкретный перечень исключений

В п. 1.2 говорится «если не оговорено отдельно» применительно к многоарендности. Есть ли уже оговорённые исключения, которые нужно включить в текущую версию?

### Q-1.2.2. Юридическая проверка — граница ответственности

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Система не выполняет «юридическую экспертизу текста иска». Где именно проходит граница?~~

**Ответ:** Проверка корректности расчётов, юридической корректности и соблюдения сроков исковой давности — **не в скоупе** текущей версии.

**Архитектурное требование:** Необходимо предусмотреть **модуль проверки корректности** (валидатор), который в текущей реализации по умолчанию возвращает ответ `«корректно»` (заглушка). Интерфейс модуля должен быть определён так, чтобы в будущем можно было подключить реальные проверки (реквизиты, госпошлина, сроки давности) без изменения вызывающего кода.

### Q-1.2.3. Будущие итерации

Есть ли предварительный роадмап того, что войдёт в v2/v3? Это нужно, чтобы архитектурные решения v1 не противоречили будущим планам (например, если в v2 планируется подача через ГАС «Правосудие», нужно учесть форматы).

### Q-1.2.4. Разграничение ответственности за ошибки

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Кто несёт ответственность за корректность данных в итоговом иске — пользователь (после ручной проверки) или система? Нужен ли дисклеймер в UI или в генерируемом документе?~~

**Ответ:** Ответственность за ошибки программы лежит на **пользователе**. Пользователь обязан проверить и при необходимости скорректировать данные перед генерацией итогового документа.

## Общие вопросы к разделу

### Q-1.G.1. Целевая аудитория

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Кто конкретный пользователь системы: штатный юрист компании, индивидуальный предприниматель без юридического образования, юридическая фирма?~~

**Ответ:** Целевая аудитория — **штатный юрист компании**. Это означает, что пользователь обладает юридической квалификацией, способен самостоятельно оценить корректность извлечённых данных и результатов расчёта. Развёрнутые юридические подсказки в UI не требуются.

### Q-1.G.2. Язык интерфейса

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Только русский? Нужна ли локализация (например, для юрлиц из стран СНГ)?~~

**Ответ:** Язык интерфейса — **только русский**. Локализация не требуется.

### Q-1.G.3. Регуляторные требования

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Подлежит ли система сертификации или аттестации (ФСТЭК, др.)? Распространяются ли на неё требования по работе с персональными данными в объёме, выходящем за рамки стандартного 152-ФЗ?~~

**Ответ:** Система **не подлежит** сертификации или аттестации (ФСТЭК и др.). Требования к защите персональных данных — в рамках стандартного 152-ФЗ, без дополнительных обременений.
