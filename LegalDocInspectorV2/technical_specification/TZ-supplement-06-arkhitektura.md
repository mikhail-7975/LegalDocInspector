# Дополнение к разделу 6. Архитектура системы — Вопросы

> Вопросы, на которые необходимо ответить для составления однозначного ТЗ.

---

## 6.1. Логические компоненты

### Q-6.1.1. Технологический стек — языки и фреймворки

> **ОТВЕТ ЗАФИКСИРОВАН**

| Компонент | Решение |
|-----------|---------|
| **Web SPA** | **React** |
| **API Gateway** | **Минимальная схема** — см. **Q-6.1.5** |
| **Оркестратор (BFF)** | **Python** |
| **Task Queue** | **Celery + RabbitMQ** |
| **ocr-pipeline** | **Python** |
| **extract-*** (извлечение полей и прочая обработка документов) | **Python** |
| **calculation-service** | **Python** (существующий калькулятор) |
| **docx-generator** | **Python** (`doc_creator` / шаблоны) |

**Итого:** весь серверный контур — **Python**, клиент — **React**. Конкретные веб-фреймворки для HTTP API оркестратора (FastAPI, Flask и т.д.) выбираются в реализации.

### Q-6.1.2. База данных

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Нужна ли СУБД для метаданных, черновиков, истории?~~

**Ответ:** **База данных не требуется.** Состояние пакета — **in-memory** в оркестраторе (**Q-6.2.1**); файлы — **файловая система** по **packageId** (**Q-6.1.4**); очередь — **RabbitMQ**; история расчётов и аудит — **не ведутся** (см. нефункциональные требования).

### Q-6.1.3. Взаимодействие между сервисами

> **ОТВЕТ ЗАФИКСИРОВАН**

~~HTTP vs очередь, service discovery?~~

**Ответ:** **Service discovery не требуется** — статические URL/хосты в конфигурации (локальный сервер, Docker Compose при необходимости). Фронтенд обращается к **HTTP API оркестратора**. Длительные шаги (OCR и связанный пайплайн) выполняются **воркерами Celery** поверх **RabbitMQ**. Синхронные операции (валидация, расчёт, генерация DOCX) вызываются из оркестратора **в том же процессе или отдельными Python-вызовами** в рамках одного деплоя — без обязательного выделения каждого шага в отдельный HTTP-микросервис для MVP.

### Q-6.1.4. Хранилище файлов — детали

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Структура каталогов, дедупликация, checksum?~~

**Ответ:** Временные и загруженные файлы хранятся в **структуре каталогов по `packageId`** (корень хранилища задаётся конфигом; внутри — вложенность по пакету и типу артефакта — на усмотрение реализации). **Дедупликация и обязательный checksum в ТЗ не задаются.** Абстракция хранилища — минимальный набор операций: запись/чтение/удаление по пути, привязанному к `packageId` и TTL (**Q-5.4.5** в дополнении по NFR).

### Q-6.1.5. API Gateway — обязателен ли?

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Отдельный Kong/Envoy или только BFF?~~

**Ответ:** Использовать **наиболее простой вариант**: **один reverse proxy** (например **Nginx** или **Caddy**) с проксированием на **Python API оркестратора** и раздачей статики **React** (или два upstream: `/` → SPA, `/api` → backend). Отдельный продукт-класса **API Gateway** (Kong, Traefik с политиками и т.д.) **не требуется**. TLS — на уровне proxy или обратного прокси перед хостом.

### Q-6.1.6. Кэширование

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Кэш OCR, шаблонов, расчётов?~~

**Ответ:** **Кэширование не требуется** (ни результатов извлечения/OCR, ни шаблонов Word, ни расчётов).

## 6.2. Диаграмма потоков

### Q-6.2.1. Управление состоянием пакета

> **ОТВЕТ ЗАФИКСИРОВАН**

~~In-memory vs Redis vs статус только в очереди?~~

**Ответ:** Состояние обработки пакета в оркестраторе ведётся **in-memory** (при перезапуске процесса оркестратора состояние активных пакетов **теряется** — согласовано с отсутствием БД и локальным сценарием).

**Модель состояний** — как в черновике ТЗ, **без упрощения**:

```
created → files_uploaded → extracting → extracted → form_editing → calculating → calculated → generating → documents_ready
```

### Q-6.2.2. Обработка частичных результатов

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Ждать всё извлечение или отдавать по мере готовности? WebSocket vs polling?~~

**Ответ:** Результаты (частичные и полные) **отдаются фронтенду по мере готовности** — не требуется блокировать ответ до завершения всех задач OCR по пакету. Обновление UI — **наиболее простым для реализации способом**: например **периодический polling** HTTP API статуса/данных пакета; **WebSocket/SSE не обязательны** для MVP.

## 6.3. Балансировка и параллелизм

### Q-6.3.1. Контейнерные ресурсы

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Лимиты CPU/RAM по контейнерам?~~

**Ответ:** Численные лимиты контейнеров в ТЗ **не фиксируются**. Ориентир — **локальный сервер уровня бытового ПК** (см. **Q-5.2.1** дополнения по NFR); `ocr-pipeline` может использовать **GPU по config** с откатом на CPU. Конкретные `cpus`/`memory` задаются при развёртывании.

### Q-6.3.2. Автомасштабирование

> **ОТВЕТ ЗАФИКСИРОВАН**

~~HPA Kubernetes?~~

**Ответ:** **Автомасштабирование не требуется.** Число процессов/воркеров (оркестратор, Celery, RabbitMQ) **задаётся вручную** в конфигурации.

### Q-6.3.3. Health checks

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Liveness / readiness / startup для каждого сервиса?~~

**Ответ:** Достаточно **стандартных health check** для компонентов, где это поддерживается из коробки: **HTTP `/health` или аналог** у API оркестратора; для **RabbitMQ** и **Celery** — типовые проверки доступности брокера и воркеров **по документации стека** без расширенной матрицы проб. Отдельные **startup probe** для тяжёлой загрузки моделей OCR — **по необходимости** в реализации, в ТЗ детализация не требуется.
