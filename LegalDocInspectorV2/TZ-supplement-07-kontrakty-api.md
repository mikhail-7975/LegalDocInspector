## Общие вопросы по API

### Q-7.1. Формат OpenAPI-спецификации

- Нужна ли полная OpenAPI 3.x спецификация до начала разработки, или разработка ведётся API-first параллельно?
- Единый файл `openapi.yaml` или по одному на каждый сервис?
- Используется ли кодогенерация клиентов/серверов из OpenAPI?

### Q-7.2. Версионирование API

- Версия только в URL-префиксе (`/api/v1`) или также в заголовке (`Accept: application/vnd.app.v1+json`)?
- Как будет происходить переход на v2 — поддержка обеих версий одновременно?

### Q-7.3. Авторизация в API

- Если аутентификация необходима (Q-5.4.1), какой механизм: Bearer token (JWT), API key, session cookie?
- Нужен ли CSRF-токен для cookie-based сессий?
- Нужен ли endpoint для аутентификации (login/logout) или это внешний IdP?

## Вопросы по конкретным эндпоинтам

### Q-7.4. `POST /packages` — Создание пакета

- Создание пакета и загрузка файлов — один запрос или два отдельных?
- Если в одном запросе — multipart/form-data с JSON-метаданными + файлами?
- Максимальный размер тела запроса?
- Какие поля обязательны при создании (только `applicationDate` или также файлы)?

### Q-7.5. `POST /packages/{packageId}/files` — Дозагрузка файла

- Как определяется «слот» файла (к какому documentSet и какому типу он относится)? Через query-параметры, form-fields?
- Можно ли перезаписать ранее загруженный файл?
- Нужен ли отдельный endpoint для удаления файла?

### Q-7.6. `POST /packages/{packageId}/extract` — Запуск извлечения

- Можно ли перезапустить извлечение для уже обработанного пакета?
- Можно ли запустить извлечение для одного конкретного documentSet или файла, а не для всего пакета?
- Ответ — синхронный (201 Accepted с task ID) или просто 202 Accepted?

### Q-7.7. `GET /packages/{packageId}/extraction` — Статус извлечения

- Какой формат статуса: `pending | processing | partial | completed | failed`?
- Включает ли ответ промежуточные результаты (частично заполненные поля по мере готовности)?
- Какова рекомендуемая частота polling (если не WebSocket)?
- Нужен ли WebSocket / SSE endpoint для push-обновлений?

### Q-7.8. `PUT /packages/{packageId}/form` — Сохранение черновика

- Это опциональный endpoint — нужен ли он для MVP?
- Если да: автосохранение по таймеру или только по кнопке?
- Конфликты: что если данные на сервере обновились (например, извлечение завершилось) — как мержить с черновиком пользователя?

### Q-7.9. `POST /packages/{packageId}/calculate` — Расчёт

- Является ли расчёт идемпотентным (одинаковые данные → одинаковый результат)?
- Сохраняется ли результат расчёта на сервере, или только возвращается клиенту?
- Если сохраняется — можно ли получить последний расчёт через GET-запрос?

### Q-7.10. `POST /packages/{packageId}/documents` — Генерация документов

- Нужно ли передавать результат расчёта в теле запроса, или сервер берёт последний сохранённый?
- Ответ — синхронный (сразу два URL) или асинхронный (202 Accepted → poll)?
- TTL ссылок на скачивание — какое конкретное значение?

### Q-7.11. `GET /artifacts/{artifactId}` — Скачивание файла

- Нужна ли авторизация для скачивания (проверка что файл принадлежит запрашивающему)?
- Поддерживается ли Range-запрос для докачки?
- Content-Disposition: `attachment` (принудительное скачивание) или `inline` (предпросмотр)?

## Формат ошибок

### Q-7.14. Полный каталог кодов ошибок

Нужен исчерпывающий перечень кодов ошибок. ТЗ упоминает формат `{ "error": { "code", "message", "details" } }`. Какие коды предусмотрены?
- `VALIDATION_ERROR` — ошибка валидации входных данных
- `EXTRACTION_FAILED` — ошибка извлечения
- `OCR_FAILED` — ошибка OCR
- `CALCULATION_ERROR` — ошибка расчёта
- `GENERATION_ERROR` — ошибка генерации документа
- `FILE_TOO_LARGE` — превышен размер файла
- `UNSUPPORTED_FORMAT` — неподдерживаемый формат файла
- `PACKAGE_NOT_FOUND` — пакет не найден
- `TIMEOUT` — превышено время ожидания
- Другие?

### Q-7.15. Локализация сообщений об ошибках

Сообщения ошибок (`message`) — всегда на русском или на английском (для машинного разбора)? Или нужны оба варианта (машинный `code` + человекочитаемый `message` на русском)?
