## 1. Назначение и границы

**Суть системы:** веб-приложение для загрузки набора документов (ЕГРЮЛ, комплекты: договор PDF, претензия PDF, справки Excel), извлечения данных, правки в форме, расчёта пени (ЖКХ/УК/ТСЖ/прочие по текущему калькулятору) и генерации двух файлов Word — **иск** и **расчёт к иску**.

**Вне объёма (базовое ТЗ):** юридическая экспертиза текста; КЭП и подача через ГАС «Правосудие»; multi-tenant и корпоративные роли (если не оговорено отдельно).

**Расчёты:** только логика [`penalty_calculator.py`](../LegalDocInspector/legal_doc_inspector/calculator/penalty_calculator.py); типы организаций УК / ТСЖ / Прочие; ставка ключевая — **из конфигурации**, не из API ЦБ; календарь выходных — допустим `isdayoff.ru` через **адаптер** для тестов.

**Шаблоны:** каталог [`data/templates/`](../data/templates/), заполнение через [`doc_creator/`](../LegalDocInspector/legal_doc_inspector/doc_creator/); два шаблона на запуск (иск + расчёт); обновление шаблонов — **config**, без админ-UI.

**Архитектура калькулятора:** интерфейс Strategy / реестр, опциональный fallback; новые типы исков — **отдельные пакеты**.

**Валидатор юридической корректности:** модуль с заглушкой «корректно», расширяемый позже.

**Ответственность:** за ошибки в итоговых документах после проверки пользователем — **пользователь** (штатный юрист организации).

**Язык UI:** только русский. **Аттестация ФСТЭК** не требуется; 152-ФЗ — см. **противоречие в §5.4**.

## 2. Роли и сценарии

**Роль пользователя:** формирует иск через веб-интерфейс.

**Администрирование:** выполняет **разработчик** (config, шаблоны, параметры расчёта, просмотр логов/очередей); **отдельного админ-UI нет**.

**Аутентификация:** **логин/пароль**; пользователь по умолчанию `admin`/`admin`; новые пользователи создаёт разработчик; **полная изоляция** данных между пользователями.

**Сессия:** при **перезагрузке вкладки** сессия **сбрасывается**, восстановление черновика **не требуется** (см. противоречие с **опциональным** `PUT .../form` в базовом ТЗ §7).

**Лимиты (config):** до **30** комплектов, до **20** справок на комплект; при превышении — ошибка с сообщением.

**Подписи комплектов:** «Комплект 1», «Комплект 2», …; переименование **не в этой итерации**.

**Загрузка:** комплекты можно добавлять постепенно; замена/удаление; после изменения состава — повторный запуск извлечения (**дельта**: только новый/изменённый комплект; удалённый — данные удалить).

**Прогресс:** по файлу — текущий файл, остаток, **оценка оставшегося времени**; недовольный OCR — **ручной ввод** (перезапуск OCR одного файла **не** предусмотрен).

**Расчёт/генерация:** многократные запуски разрешены; предыдущие сгенерированные файлы на сервере **не хранятся**; история расчётов **не ведётся**.

**Таймауты извлечения:** **нет** жёсткого таймаута.

**Ошибки извлечения:** пустые поля + сообщение у блока (FR-7); «битый файл» — отдельное сообщение + пустые поля.

**Обязательные поля формы (расчёт):** перечень как в `streamlit/interface.py` (истец, ответчик, суд, договоры, иск, `responsitive_name` и т.д.) — **все обязательны**; незаполненное — подсветка и **блокировка** «Произвести расчёт».

**Клиент:** только **десктоп**; мобильная вёрстка **не требуется**.

### Противоречия / уточнения

| Тема | Версия A | Версия B |
|------|----------|----------|
| Сохранение черновика формы | Базовое Т §7: `PUT /packages/{id}/form` опционально | **Доп. 2:** сессия теряется при перезагрузке; явного требования серверного черновика нет — **для MVP согласовать**: либо не реализовывать PUT, либо ограничить сценарий «одна вкладка без перезагрузки» |

## 3. Входные данные

Логическая модель пакета — как в базовом Т **§3.1** (`packageId`, `applicationDate`, `egrulExtract`, `documentSets[]` с `contract`, `preClaim`, `debtCertificates[]`).

**Справка Excel:** основной источник для расчёта; парсер — `exel_parser.TableParser`; формат фиксированный (данные с **строки 12**, лист **0**); **без формул**; ошибка чтения/расчёта — сообщение «Невозможно произвести рассчёты: … Произведите рассчёты вручную» (формулировка из доп. 3).

**Договор PDF:** тип ТЭ/ГВС/СОИ/ФОТЭ, номер, день оплаты, пункт договора; просрочка = срок + 1 день; если срок не найден — подсказка «введите вручную».

**Претензия PDF:** дата и номер; иначе «данные не найдены» + ручной ввод.

**ЕГРЮЛ:** поля компании; типичный вход — **скачанный с сайта ФНС**; **проверка свежести выписки не выполняется**.

**Конфликты полей между документами:** по каждому документу показывать извлечённые значения; при расхождении — «Обнаружены различия в данных в полях …» + список полей; итог пользователь вводит/выбирает вручную.

**PDF:** постраничный OCR; ранний выход, когда все обязательные поля извлечены; частично извлечённые значения **сохранять** и продолжать страницы.

**Скан:** DPI **не нормируется**; допускаются любые сканы и **фото с телефона**.

**Язык документов:** только **русский**.

**Файлы:** должны открываться и читаться; иначе уведомление о невозможности открытия; парольные PDF/Excel — то же.

**Дата заявления:** вводит пользователь; по умолчанию **сегодня**; доп. валидация дат **не задана**.

## 4. Функциональные требования

**FR-1 (загрузка):** данные собираются на **фронтенде**, передача на backend **одним пакетом**; перед закрытием/перезагрузкой страницы — **`beforeunload`** с предупреждением о потере данных.

**Обрыв загрузки PDF:** повторная загрузка **только** оборвавшегося файла; протоколы resumable (tus и т.д.) **не нужны**.

**UI загрузки:** **drag-and-drop** и **file input**.

**Сообщения валидации:** русский язык; тексты в **config-файле**.

**Извлечение PDF:** интерфейс **OCR-движка** + интерфейс **извлекателя полей**; реализации выбираются **config**; смена движка — минимальными правками.

**Многостраничные PDF:** см. §3 (постранично, частичные результаты, стоп при полноте).

**Расчёт:** `penalty_calculator.py`; ставка ЦБ в версии продукта — **неизменна на весь период**; валюта — **рубли**; **отдельный расчёт на каждый комплект** (без обязательной консолидации в одно число).

**Валидация:** обязательна на **клиенте перед генерацией**; целесообразна и перед расчётом; сервер не исключать архитектурно.

**Генерация:** как в `doc_creator` + шаблоны `data/templates/`.

**Скачивание:** **два отдельных файла**; **два нажатия** (по одному на файл); **без ZIP**.

**Имена файлов (доп. 4 Q-4.4.5):** шаблон `isk_<packageId>.docx` и `calculation_<packageId>.docx` (в дополнении сокращённо; в базовом Т **FR-12** в одном месте фигурирует `isk_<packageId>.docx`, `raschet_<packageId>.docx`).

## 5. Нефункциональные требования

**Производительность OCR:** **верхняя граница времени в ТЗ не задаётся** (в базовом Т были ориентиры 10–15 мин — см. противоречие ниже).

**UX:** **оценка времени OCR** с учётом очереди; push — на **фронте** (Browser Push и т.п.); единая **панель текстовых уведомлений** от backend.

**Пользователи:** число сессий — **ориентир** (базовое Т: до 5); допускается рост очереди при большем числе пользователей; **рост нагрузки не планируется**.

**Инфраструктура:** **локальный сервер**, мощность **уровня бытового ПК**; не требуется облако/Kubernetes в базовом объёме дополнения.

**GPU:** включается из **config**; нет GPU — расчёт/OCR на **CPU**.

**Логи:** **stdout** + **journald**; формат **plaintext**; поля: **timestamp**, **level**, **service_name**, **стадия**; `correlation_id` **не обязателен** (противоречие базовому **NFR-7**).

**Алерты:** внешняя система алертов **не входит**; админ смотрит логи.

**Падение ocr-pipeline:** уведомление о ручном вводе; следующая задача; **без retry и без DLQ**.

**Безопасность:** **HTTPS** в проде (Т NFR-9); логин/пароль; **шифрование at rest не требуется**; **журнал аудита не требуется**.

**TTL временных файлов:** **8 часов** (дополнение **явно заменяет** диапазон 24–72 ч из базового Т **NFR-11**).

**Клиент:** десктоп; мобильная адаптация **не требуется** (см. также Т **NFR-12** про поддержку браузеров — для десктопных версий).

## 6. Архитектура

**Зафиксировано дополнениями:**

- **Стек:** **React** (SPA), **Python** (оркестратор, OCR-контур, расчёт, генерация), **Celery + RabbitMQ**.
- **БД:** **не используется**; состояние пакета — **in-memory** (потеря при рестарте оркестратора).
- **Файлы:** каталоги по **`packageId`**; TTL см. §5.
- **Gateway:** простой **reverse proxy** (Nginx/Caddy) + статика React + API.
- **Кэш:** **не требуется**.
- **Service discovery:** **не требуется**.
- **Результаты извлечения:** отдача **по мере готовности**; обновление UI — **polling** (простой вариант); WebSocket/SSE **не обязательны**.
- **Состояния пакета:** `created → files_uploaded → extracting → extracted → form_editing → calculating → calculated → generating → documents_ready`.
- **Автомасштабирование:** **не требуется**.
- **Health checks:** стандартные (`/health` у API, проверки брокера/воркеров по документации).

### Противоречия / уточнения

| Тема | Версия A (базовое Т §5–6) | Версия B (Доп. 5–6) |
|------|---------------------------|---------------------|
| Деплой | Отдельные контейнеры **каждого** extract-\*, масштабирование **ocr-pipeline** первым, ≥2 реплик OCR, общая очередь | **Локальный ПК**, **нет K8s/HPA**, воркеры вручную; допускается **монолитный** Python с Celery без обязательных отдельных HTTP-микросервисов на каждый extract |
| BFF | Только маршрутизация без тяжёлой обработки | Синхронные расчёт/DOCX **в процессе оркестратора** возможны для MVP |
| Очередь | RabbitMQ или Redis Streams (Т) | **Celery + RabbitMQ** (зафиксировано) |

**Итог для реализации:** заказчик ориентируется на **версию B** (дополнения); текст базового Т §6 описывает **целевую распределённую** картину — при портировании на локальный сервер **версия B** имеет приоритет.
