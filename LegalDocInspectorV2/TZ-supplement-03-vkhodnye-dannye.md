## 3.1. Состав набора документов

### Q-3.1.4. Справка о задолженности (Excel) — ОСНОВНОЙ ИСТОЧНИК РАСЧЁТА

> **ОТВЕТ ЗАФИКСИРОВАН**

**Извлекаемые поля:**
- ИНН задолжника
- Период задолженности (набор месяцев в формате `MM.YYYY`)
- Сумма задолженности по месяцам
- Сумма оплат
- Даты оплат
- Информация о корректировках

Это **основной источник данных** для расчёта пени — именно из справки берётся помесячная разбивка задолженности, оплаты и корректировки, которые передаются в `penalty_calculator.py`.

**Требование к файлу:** справка в Excel предоставляется **без формул** — в ячейках должны быть сохранены **готовые числовые и текстовые значения** (не вычисления по формулам).

Если при чтении или разборе справки возникает ошибка, либо данные извлечь не удалось, либо в автоматических расчётах произошла ошибка, пользователю отображается сообщение:

«Невозможно произвести рассчёты: » + **текст ошибки** + «. Произведите рассчёты вручную»

(текст ошибки — сообщение из исключения, парсера или движка расчёта), плюс в интерфейсе — возможность внести результат расчёта вручную.

- Если срок оплаты **не найден**, возвращается текст "введите вручную" и в интерфейсе отображается уведомление рядом с полем для ввода

### Q-3.1.5. Версия готового парсера Excel

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Какой именно «готовый парсер»? Привязан ли к конкретному формату?~~

**Ответ:** Используется парсер из `LegalDocInspector/legal_doc_inspector/exel_parser.py` — класс `TableParser` (на базе `ExcelReader` / `pandas`).

**Парсер привязан к конкретному формату справки** конкретной организации. Особенности формата:
- Данные начинаются со строки 12 (строки 0–11 — шапка).
- Чтение с первого листа (`sheet_name=0`).
- Распознавание строк по типам:
  - Заголовок месяца (паттерн: `январь 2024`, `февраль 2025` и т.д.)
  - Блок годовой корректировки (`доля от размера годовой корректировки платы за тепловую энергию`)
  - Итого по договору / итого по периоду
  - Строки начислений (период `MM.YYYY` + сумма)
  - Строки оплат (дата + сумма + тип договора)
  - Строки задолженности и итогов

**Выход парсера:** словарь, сгруппированный по месяцам, с подблоками `accrual` / `adjustment`, каждый содержит `accruals`, `additionals`, `payments`, `debt`. Этот формат напрямую подаётся в `penalty_calculator.calculate_penalty()`.

Парсер **не универсальный** — при изменении формата справки потребуется адаптация. Архитектурно рекомендуется выделить парсер за интерфейс (Strategy), чтобы в будущем можно было подключить парсер для другого формата справки.

### Q-3.1.6. Соответствие данных из разных документов

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Как должна вести себя система, если данные из разных документов противоречат друг другу?~~

**Ответ:**

1. **Отображение по документам.** Для каждого документа в интерфейсе показываются поля с данными, извлечёнными **из этого документа** (пользователь видит источник значения).

2. **Конфликты.** Если одно и то же поле получено из нескольких документов и **значения различаются**, в интерфейсе отображается надпись:

   **«Обнаружены различия в данных в полях …»**

   и **перечисляются названия полей**, в которых данные не совпадают.

3. **Разрешение конфликта** — пользователь вручную выбирает или вводит итоговое значение (как часть общего сценария редактирования формы до расчёта и генерации).

### Q-3.2.2. Количество страниц в PDF

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Есть ли ограничение на количество страниц в одном PDF?~~

**Ответ:**

Жёсткого лимита числа страниц в ТЗ не задаётся. Обработка PDF строится так:

1. Страницы обрабатываются **последовательно** (по порядку): каждая страница переводится в текст с помощью **OCR-движка**.
2. После получения текста очередной страницы система **пытается извлечь** все **нужные для комплекта** данные из уже накопленного текста (текущая страница + предыдущие при необходимости логики извлечения).
3. Как только извлечение признано **успешным** (все обязательные поля для данного типа документа получены с достаточной уверенностью / по правилам парсера), **дальнейший OCR оставшихся страниц не выполняется** — перевод PDF в текст прекращается (ранний выход).
4. Если после прохода по всем страницам данные не извлечены полностью — дальше действует общий сценарий: пустые поля и ручной ввод (**Q-2.3.2**, **Q-2.3.3** в приложении по ролям и сценариям).

Таким образом, для длинных договоров (100+ страниц) время OCR ограничивается тем, на какой странице окажутся нужные сведения, а не полным сканированием документа.

### Q-3.2.3. Качество сканов

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Минимальное DPI, цветность, штампы, фото с телефона?~~

**Ответ:**

- **Минимальное допустимое разрешение (DPI)** для гарантированного успеха OCR в ТЗ **не фиксируется** (ограничение неизвестно / не нормируется на стороне продукта).
- Входом считаются **любые сканы и снимки**, в том числе **сфотографированные на телефон**; допускаются цветные, ч/б и оттенки серого, наличие штампов, печатей и рукописных пометок — качество распознавания определяется возможностями OCR-движка, без отдельных порогов в ТЗ.

### Q-3.2.4. Язык документов

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Только русскоязычные документы или двуязычные договоры?~~

**Ответ:** **Договоры и иные документы набора — только на русском языке.** Двуязычные (русский/английский) договоры в объём текущего ТЗ не входят.

### Q-3.2.5. Защита файлов и доступность для чтения

> **ОТВЕТ ЗАФИКСИРОВАН**

~~PDF с паролем, защищённый Excel, макросы?~~

**Ответ:**

1. **Общее правило:** загруженный файл должен **открываться** используемыми в системе средствами и быть **доступен для чтения** содержимого, необходимого для извлечения данных. Если открытие или чтение **невозможны**, пользователю показывается **уведомление о невозможности открытия (чтения) файла** с указанием причины, насколько она доступна из средства обработки файла.
2. **PDF с паролем на открытие**, **Excel, защищённый от чтения паролем**, иные случаи, когда библиотека не может получить данные, — трактуются как **невозможность открытия/чтения** (уведомление по п. 1).
3. **Справки Excel:** требование **без формул** и сообщение при ошибках чтения/расчёта — см. **Q-3.1.4**.
4. **Макросы и формулы:** типовая справка — значения без формул (**Q-3.1.4**). Файлы, где итоговые числа недоступны без выполнения макросов, не считаются корректным входом; при ошибке чтения — как в **Q-3.1.4**.

### Q-3.2.6. Выписка ЕГРЮЛ — источник и формат

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Электронная vs бумажная, проверка даты выписки?~~

**Ответ:**

- **Типовой формат:** выписка ЕГРЮЛ, **скачанная с сайта ФНС** (электронный документ с текстовым слоем; при необходимости извлечения полей используется тот же подход, что и для прочих PDF в системе).
- **Проверка актуальности выписки** (ограничение по давности даты выписки и т.п.) **не выполняется** — ответственность за актуальность на стороне пользователя.

### Q-3.2.7. Дата составления заявления

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Когда заполняется, значение по умолчанию, валидация?~~

**Ответ:**

- Поле **заполняет пользователь** в форме (до расчёта и генерации документов).
- **Значение по умолчанию:** **сегодняшняя дата** (текущий календарный день); пользователь может изменить.
- **Дополнительная валидация** даты (например, запрет даты в глубоком прошлом) в ТЗ **не задаётся**.
