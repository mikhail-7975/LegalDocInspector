# Дополнение к разделу 2. Роли и пользовательские сценарии — Вопросы

> Вопросы, на которые необходимо ответить для составления однозначного ТЗ.

---

## 2.1. Роли

### Q-2.1.1. Роль администратора

> **ОТВЕТ ЗАФИКСИРОВАН**

~~В ТЗ определена единственная роль «Пользователь». Кто управляет системой?~~

**Ответ:** Роль администратора выполняет **разработчик системы**:
- Загрузка/обновление шаблонов Word — через config-файлы.
- Настройка параметров расчётов (ставки, периоды) — через config-файлы.
- Мониторинг состояния очередей и сервисов — разработчиком напрямую (логи, дашборды инфраструктуры).

**Отдельный интерфейс администратора не требуется.** Вся административная работа ведётся через конфигурационные файлы и инструменты инфраструктуры.

### Q-2.1.2. Аутентификация и идентификация

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Нужна ли аутентификация пользователей вообще? Каков механизм?~~

**Ответ:** Аутентификация **нужна** — для организации работы нескольких пользователей одновременно и изоляции данных.

- **Механизм:** только логин/пароль (без SSO, OAuth и т.д.).
- **Пользователь по умолчанию:** логин `admin`, пароль `admin`.
- **Добавление пользователей:** осуществляет разработчик (через конфиг / БД / CLI — не через UI).
- Регистрация через интерфейс не предусмотрена.

### Q-2.1.3. Изоляция данных между пользователями

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Может ли один пользователь видеть пакеты другого? Нужна ли строгая изоляция данных?~~

**Ответ:** Данные между пользователями **полностью изолированы**. Пользователь видит только свои пакеты.

## 2.2. Основной сценарий

### Q-2.2.1. Сохранение и восстановление сессии

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Может ли пользователь закрыть браузер и вернуться к ранее созданному пакету?~~

**Ответ:** Сессия **полностью сбрасывается при перезагрузке вкладки**. Возврат к незавершённому пакету не предусмотрен — пользователь начинает работу заново.

### Q-2.2.2. Лимит комплектов документов

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Есть ли верхняя граница на количество `documentSets`? Есть ли верхняя граница на количество справок?~~

**Ответ:**
- Максимум **до 30 комплектов** (`documentSets`) в одном пакете — параметр конфигурируется.
- Максимум **до 20 справок о задолженности** (`debtCertificates`) в одном комплекте — параметр конфигурируется.
- При превышении любого из лимитов отображается **сообщение об ошибке** с текстом ошибки.

### Q-2.2.3. Именование комплектов (label)

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Кто задаёт `label`? Можно ли переименовать?~~

**Ответ:** Комплекты именуются автоматически: **«Комплект 1»**, **«Комплект 2»**, **«Комплект 3»** и т.д. (сквозная нумерация). Переименование **не предусмотрено** в текущей итерации; в следующих версиях возможно добавление изменения названия комплекта.

### Q-2.2.4. Порядок загрузки файлов

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Обязан ли пользователь загрузить все файлы за одну сессию? Можно ли заменить/удалить? Можно ли добавить комплект после извлечения?~~

**Ответ:**
- Пользователь может **добавлять комплекты постепенно** (не обязательно все за один раз).
- **Замена и удаление** загруженного комплекта — доступны.
- **Дополнительный комплект можно добавить после завершения извлечения.** В этом случае необходимо заново запустить извлечение. Система проверяет изменения в составе комплектов:
  - Если комплект **добавлен** — извлечение выполняется **только для нового** комплекта (ранее извлечённые данные сохраняются).
  - Если комплект **удалён** — извлечённые данные этого комплекта **удаляются**.
  - Если комплект **не изменился** — повторное извлечение **не выполняется**.

### Q-2.2.5. Повторный запуск извлечения

> **ОТВЕТ ЗАФИКСИРОВАН (следует из Q-2.2.4)**

~~Может ли пользователь перезапустить извлечение для одного комплекта? Нужен ли ручной перезапуск?~~

**Ответ:** Перезапуск извлечения инициируется пользователем вручную (кнопка «Запустить извлечение»). Система детектирует изменённые/добавленные/удалённые комплекты и обрабатывает только дельту (Q-2.2.4). Если пользователь заменил файл в комплекте — этот комплект считается изменённым и извлечение для него выполняется заново.

### Q-2.2.6. Индикация прогресса

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Какой уровень детализации прогресса?~~

**Ответ:** Пользователю отображается прогресс **по каждому файлу**:
- Какой файл сейчас анализируется.
- Сколько файлов осталось.
- Оценка оставшегося времени.

Если пользователь недоволен результатами OCR — он вводит данные вручную (перезапуск OCR для отдельного файла не предусмотрен).

### Q-2.2.7. Множественные расчёты и генерации

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Может ли пользователь выполнить расчёт/генерацию несколько раз?~~

**Ответ:**
- **Расчёт** — пользователь может выполнять **несколько раз**, меняя входные данные между расчётами.
- **Генерация документов** — пользователь может генерировать **несколько раз**. Предыдущие генерации **не сохраняются** (перезаписываются).
- **История** расчётов/генераций для одного пакета **не требуется**.

### Q-2.2.8. Валидация формы перед расчётом

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Какие поля обязательны? Что происходит, если не заполнены?~~

**Ответ:** Поля формы для расчёта реализованы в `LegalDocInspector/streamlit/interface.py`. Перечень полей:

**Глобальные параметры:**
- Дата конца просрочки (`end_date`)
- Тип компании (`company_type`: Прочие / УК / ТСЖ)

**Данные о суде:**
- Название органа (`court_info.name`)
- Адрес органа (`court_info.addres`)

**Данные об истце (`plaintiff_info`):**
- ИНН
- Полное наименование
- Сокращённое наименование
- Адрес
- Адрес для корреспонденции
- ОГРН

**Данные об ответчике (`defendant_info`):**
- Полное наименование
- Сокращённое наименование
- Адрес
- ИНН
- ОГРН

**По каждому договору (`contracts`):**
- Тип договора (ТЭ / ГВС / СОИ / ФОТЭ)
- День месяца — последний день оплаты (`day_of_penalty`)
- Номер пункта договора о дне начала просрочки (`contract_point`)

**Данные об иске:**
- Цена иска (`lawsuit_info.cost`)
- Госпошлина (`lawsuit_info.tax`)
- Список претензий (номер и дата)
- ФИО ответственного лица (`responsitive_name`)

**Все поля обязательны.** Если хотя бы одно поле не заполнено — оно подсвечивается и **кнопка «Произвести расчёт» блокируется**.

## 2.3. Альтернативные сценарии

### Q-2.3.1. Конкретные таймауты

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Каков точный таймаут ожидания извлечения?~~

**Ответ:** Конкретных таймаутов **нет**. Система ожидает завершения извлечения без жёсткого ограничения по времени. Пользователь видит прогресс (Q-2.2.6) и может ориентироваться на оценку оставшегося времени.

### Q-2.3.2. Поведение при полном отказе

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Что происходит, если извлечение не удалось ни для одного документа?~~

**Ответ:** При ошибке извлечения данных из документа возвращаются **пустые поля**. Пользователю необходимо заполнить их **вручную**. Это распространяется как на частичный отказ (один файл), так и на полный отказ (все файлы). Специальное сообщение об ошибке отображается рядом с соответствующим блоком полей (согласно ТЗ, FR-7).

### Q-2.3.3. Обработка «битых» файлов

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Что если загруженный PDF повреждён и его невозможно даже открыть?~~

**Ответ:** Если файл невозможно открыть — отображается **ошибка с сообщением о битом файле** и пользователю предоставляются **пустые поля для ввода данных вручную**. Логика аналогична Q-2.3.2 (ошибка извлечения → пустые поля), но с отдельным текстом ошибки, указывающим на повреждённый файл.

### Q-2.3.4. Мобильные устройства

> **ОТВЕТ ЗАФИКСИРОВАН**

~~Должен ли UI корректно работать на мобильных устройствах / планшетах, или только десктоп?~~

**Ответ:** UI работает **только для десктопа**. Адаптивная вёрстка для мобильных устройств и планшетов не требуется.
