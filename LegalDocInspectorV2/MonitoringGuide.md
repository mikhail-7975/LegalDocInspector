# Руководство по мониторингу сервиса (для разработчика / администратора)

Документ описывает, **как разработчику следить за работой** LegalDocInspector при развёртывании по [`technical_specification/FullSpecification.md`](technical_specification/FullSpecification.md) и целевой структуре из [`ProjectStructure.md`](ProjectStructure.md).

**Важно по ТЗ:** отдельный продуктовый мониторинг (Prometheus, Grafana, ELK) **не требуется** для MVP; автоматические алерты (PagerDuty, почта) **не входят в объём** — контроль ведётся через **логи** и при необходимости **ручные проверки** health и очередей.

---

## 1. Что отслеживать (сводка)

| Область | Что смотреть | Источник |
|--------|----------------|----------|
| HTTP API | Доступность, коды ответов, задержки | Логи reverse proxy / приложения, `GET /health` |
| Очередь | Накопление задач, отсутствие воркеров | RabbitMQ management, логи Celery |
| Пайплайн | Стадии пакета, ошибки извлечения/расчёта | Логи приложения (поле **стадия**) |
| Файлы | Заполнение диска, TTL временных файлов | `df`, конфиг TTL (FullSpecification § 5), каталог хранилища |
| Внешние вызовы | Сбои календаря выходных (`isdayoff` / аналог в легаси-калькуляторе) | Логи при расчёте пени, сетевые ошибки |
| Пользовательский UX | Сообщения в UI не заменяют логи сервера | Панель уведомлений + логи бэкенда при расследовании |

---

## 2. Логи (основной инструмент)

### 2.1. Требования ТЗ

- **Вывод:** **stdout**, на типичном Linux-хосте — приём **journald** (см. FullSpecification § 5, дополнение по NFR).
- **Формат:** **plaintext** (не обязательно JSON).
- **Поля в строке (явно или фиксированным префиксом):** **timestamp**, **level**, **service_name**, **стадия** обработки (этап пайплайна: загрузка, OCR, извлечение, расчёт, генерация и т.д.).
- **`correlation_id` / `X-Request-Id`:** **не обязательны**; для отладки можно ввести опционально (см. противоречие NFR-7 vs Доп. 5 в FullSpecification).

### 2.2. Практика для разработчика

- **Docker Compose:** `docker compose logs -f api`, `docker compose logs -f worker` (имена сервисов заменить на фактические из `infra/docker-compose.yml`).
- **systemd + journald:** `journalctl -u <имя-сервиса> -f` или фильтр по unit-файлам API и Celery worker.
- При поиске инцидента по пользователю ориентироваться на **`packageId`** и стадию в сообщениях (если реализация их пишет).

## 3. Проверки здоровья (health checks)

По FullSpecification § 6:

- HTTP **`GET /health`** у API оркестратора — базовая проверка «процесс жив».
- **RabbitMQ** и **Celery workers** — проверки «по документации» выбранных инструментов (например, открыт ли порт брокера, отвечает ли management API RabbitMQ, есть ли активные consumer’ы у очереди OCR/извлечения).

**Рекомендуемый ручной чеклист после деплоя:**

1. `curl -sf http://localhost:<порт>/health` (или через HTTPS за reverse proxy) → ожидается успех.
2. Очередь RabbitMQ: нет бесконечного роста необработанных сообщений при простое без нагрузки.
3. Воркер Celery: в логах нет циклических падений при старте (импорты, брокер, путь к легаси-модулям).

## 4. Компоненты архитектуры и что смотреть

Согласно FullSpecification § 6 и [`ProjectStructure.md`](ProjectStructure.md):

| Компонент | Симптомы проблемы | Действия |
|-----------|-------------------|----------|
| **Reverse proxy** (Nginx/Caddy) | 502/504, нет статики SPA | Логи proxy, доступность upstream `api` |
| **Оркестратор (Python API)** | 5xx, таймауты | Логи приложения, `/health`, нагрузка CPU/RAM |
| **Celery + RabbitMQ** | Задачи не берутся, очередь растёт | `celery inspect active`, состояние брокера, логи worker |
| **ocr-pipeline / извлечение** | Долгий OCR, ошибки движка | Логи стадии `OCR` / `extracting`, конфиг GPU/CPU (§ 5.2.2 доп.) |
| **Легаси `penalty_calculator`** | Ошибки при расчёте пени | Логи вызова; сетевые сбои к сервису выходных дней (не править ядро — см. TZ-supplement-12) |
| **Файловое хранилище** | Запись не удаётся | Права на `STORAGE_ROOT`, место на диске, TTL |

**Состояния пакета** (для сверки с логами):  
`created → files_uploaded → extracting → extracted → form_editing → calculating → calculated → generating → documents_ready` (FullSpecification § 6).

## 5. Чего не ждать по ТЗ

- **Нет** обязательной интеграции с ELK/Grafana/Prometheus для MVP.
- **Нет** встроенной системы алертов на почту/мессенджер — администрирование через **просмотр логов** (FullSpecification § 5).
- **Нет** жёсткого численного SLA по времени OCR — «верхняя граница не задаётся» (Доп. 5 Q-5.0.1); деградация проявляется в **росте очереди** и **ETA в UI**, а не в автоматическом алерте.

---

## 6. Расследование типичных инцидентов

1. **Пользователь видит ошибку в панели уведомлений** — найти ту же ошибку в логах API/worker по времени и `packageId` (если логируется).
2. **Расчёт «Невозможно произвести рассчёты»** — проверить логи вокруг `TableParser` / `calculate_penalty` (адаптеры в новом коде должны перехватывать исключения и отдавать сообщение в API).
3. **502 от nginx** — upstream API не запущен или упал; проверить `health` и перезапуск контейнера/процесса.
4. **Очередь OCR не двигается** — воркер не запущен или зависла задача; логи worker, при необходимости перезапуск воркера (осознавая потерю in-memory состояния при рестарте оркестратора — FullSpecification § 6).
