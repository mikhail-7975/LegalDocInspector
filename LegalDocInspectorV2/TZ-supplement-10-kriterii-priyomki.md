# Дополнение к разделу 10. Критерии приёмки — Вопросы

> Вопросы, на которые необходимо ответить для составления однозначного ТЗ.

---

## Общие вопросы по приёмке

### Q-10.1. Кто проводит приёмку?

- Заказчик лично?
- Выделенный QA-специалист?
- Совместная приёмка (заказчик + разработчик)?
- Какова процедура (акт приёмки, протокол тестирования)?

### Q-10.2. Среда для приёмки

- Приёмка на стенде разработчика, на стенде заказчика, или на prod-like окружении?
- Кто развёртывает стенд для приёмки?
- Какие аппаратные ресурсы доступны для приёмочного тестирования?

### Q-10.3. Тестовые данные для приёмки

- Кто предоставляет реальные документы для тестирования (договоры, претензии, справки, ЕГРЮЛ)?
- Допускаются ли синтетические (фейковые) документы?
- Сколько комплектов документов нужно успешно обработать для приёмки?
- Нужны ли «негативные» тестовые кейсы (битые файлы, неподдерживаемые форматы, пустые файлы)?

## Вопросы по конкретным критериям

### Q-10.4. Критерий 1 — Сквозной сценарий

«От начала до скачивания двух DOCX без обращения к разработчику»:
- На каких конкретных документах? Нужен набор эталонных тестовых файлов.
- Нужно ли проверить, что содержимое DOCX корректно (правильные данные в нужных местах), или достаточно проверки что файл открывается?
- Кто определяет «корректность» содержимого — сверка с эталонным образцом?

### Q-10.5. Критерий 2 — Ошибка извлечения

«При симуляции ошибки извлечения»:
- Как именно симулируется ошибка — подача заведомо нечитаемого файла, или намеренная деградация сервиса?
- Достаточно ли проверить один тип ошибки или несколько (ошибка OCR, ошибка парсера Excel, timeout)?

### Q-10.6. Критерий 4 — Валидность DOCX

«Два файла, открываемых в Microsoft Word / LibreOffice без ошибок структуры»:
- Какая версия Word считается эталонной (2019, 2021, 365)?
- Нужно ли проверять форматирование (шрифты, отступы, нумерация) или только открываемость?
- Должен ли документ корректно печататься?

### Q-10.7. Критерий 5 — Нагрузочный сценарий

«5 параллельных полных циклов»:
- Каков точный сценарий нагрузочного теста (последовательность действий каждого виртуального пользователя)?
- Какие документы используются (одинаковые для всех 5 сессий или разные)?
- Каковы конкретные пороги pass/fail по времени отклика?
- Какой инструмент для нагрузочного тестирования (k6, JMeter, Locust)?

### Q-10.8. Критерий 7 — OCR

«Извлечение полей завершается успешно за счёт OCR-пайплайна»:
- Какой процент полей должен быть заполнен для считания OCR «успешным»?
- На каком качестве скана проверять (300 DPI чёрно-белый, 150 DPI цветной, фото с телефона)?
- Нужны ли эталонные ожидаемые значения (ground truth) для сверки?

### Q-10.9. Критерий 10 — Локальное хранилище

«Без обязательной зависимости от внешнего объектного хранилища»:
- Достаточно ли демонстрации на Docker Compose с монтированием тома?
- Нужно ли проверять очистку по TTL?
- Нужно ли проверять работу при нехватке дискового пространства?

## Дополнительные критерии

### Q-10.10. Критерий производительности по типам операций

Нужны ли отдельные SLA по критериям:
- Время загрузки файлов?
- Время извлечения (без OCR)?
- Время расчёта?
- Время генерации DOCX?
- Или достаточно общего сквозного SLA?

### Q-10.11. Регрессионное тестирование

Нужны ли автоматические регрессионные тесты, которые можно запускать при каждом обновлении, или достаточно ручной приёмки?

### Q-10.12. Этапность приёмки

Приёмка всей системы целиком или поэтапная (например: сначала загрузка + извлечение, потом расчёт, потом генерация)?
