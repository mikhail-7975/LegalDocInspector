# Дополнение к разделу 5. Нефункциональные требования — Вопросы

> Вопросы, на которые необходимо ответить для составления однозначного ТЗ.

---

## 5.0. Профиль производительности

### Q-5.0.1. Конкретные SLA по времени OCR

ТЗ говорит что OCR — «наиболее затратно по времени», но не даёт конкретных цифр.
- Каково ожидаемое время OCR одной страницы PDF?
- Каково ожидаемое время OCR типового договора (например, 10 страниц)?
- Каково максимально допустимое время ожидания пользователем результата извлечения **одного комплекта**?

### Q-5.0.2. Прогресс-бар или статическое ожидание

Ориентир «10–15 мин на полный пакет при пиковой очереди OCR» — это нормально для UX? Нужен ли push-уведомление (email, websocket), чтобы пользователь мог уйти со страницы и вернуться?

## 5.1. Одновременная работа

### Q-5.1.1. Число пользователей — жёсткий лимит или ориентир?

5 одновременных сессий (NFR-1) — это:
- **Жёсткий лимит**: 6-й пользователь получает ошибку «система занята»?
- **Ориентир**: система проектируется на 5, но при 6-7 допускается деградация (увеличение времени ожидания)?

### Q-5.1.2. Планируемый рост нагрузки

Предполагается ли рост числа одновременных пользователей после MVP? Если да — до какого количества и в какие сроки? Это влияет на выбор инфраструктуры.

## 5.2. Масштабирование

### Q-5.2.1. Целевая инфраструктура развёртывания

Куда будет развёрнута система?
- Собственный сервер (bare metal / VM)?
- Облако (Yandex Cloud, VK Cloud, Selectel, AWS)?
- Kubernetes или Docker Compose?
- Каковы доступные ресурсы (CPU, RAM, GPU для OCR)?

### Q-5.2.2. GPU для OCR

Планируется ли использование GPU для ускорения OCR? Если да — какие карты доступны? Если нет — OCR на CPU, что существенно медленнее.

## 5.3. Надёжность и наблюдаемость

### Q-5.3.1. Стек логирования и мониторинга

Какой стек используется для логов и мониторинга?
- ELK (Elasticsearch + Logstash + Kibana)?
- Grafana + Loki + Prometheus?
- Простой вывод в stdout + journald?
- Коммерческий SaaS (Datadog, New Relic)?

### Q-5.3.2. Формат логов

- Структурированные JSON-логи?
- Plaintext?
- Какие обязательные поля в каждой записи (timestamp, level, correlation_id, service_name, ...)?

### Q-5.3.3. Алерты

Нужны ли автоматические алерты при:
- Очередь OCR превышает N задач?
- Среднее время обработки превышает SLA?
- Ошибка 5xx у сервиса?
- Нехватка места на диске для хранилища?

### Q-5.3.4. Отказоустойчивость при падении сервиса

Что происходит, если один из сервисов (например, `ocr-pipeline`) упал посреди обработки задачи?
- Задача автоматически повторяется (retry из очереди)?
- Сколько попыток? С каким интервалом?
- Какой механизм dead letter queue?

## 5.4. Безопасность

### Q-5.4.1. Аутентификация — уточнение

В открытых вопросах (раздел 11) упомянуто: «без авторизации / базовая / SSO». Нужно решение:
- Для MVP — без аутентификации (любой с доступом к URL может пользоваться)?
- Или базовая аутентификация (login/password)?
- Или интеграция с корпоративной SSO?

### Q-5.4.2. Шифрование данных at rest

Нужно ли шифровать файлы в хранилище (at rest), или достаточно защиты на уровне файловой системы / сети?

### Q-5.4.3. Аудит доступа

Нужен ли журнал аудита (кто, когда, к какому пакету обращался, какие действия выполнял)?

### Q-5.4.4. 152-ФЗ и персональные данные

- Какие именно персональные данные обрабатываются (ФИО руководителей, адреса)?
- Нужно ли согласие на обработку персональных данных от пользователя?
- Где физически будут храниться данные (требование к территории РФ)?
- Срок хранения персональных данных?

### Q-5.4.5. TTL временных файлов — точное значение

ТЗ указывает диапазон «24–72 ч». Каково точное значение TTL?
- Для исходных загруженных файлов?
- Для результатов OCR?
- Для сгенерированных DOCX?
- Одинаковый TTL для всех или разный?
